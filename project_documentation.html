1. Project Overview

bolt.diy is an AI-powered full-stack web development platform that runs entirely in the browser. It provides a conversational interface where users describe what they want to build, and the AI generates complete codebases, manages files, runs terminals, and deploys projects — all from within a sandboxed environment.

The platform consists of three main applications:

🖥️ Desktop Frontend

Rich single-page application with code editor, terminal emulator, preview pane, Git integration, and deployment UI. Built with React 18 + UnoCSS.

📱 Mobile Frontend

Native Android app via Capacitor with chat, file explorer, terminal, deploy, and settings screens. Built with React 19 + Tailwind CSS.

⚙️ Backend API

Express.js server with 26+ REST API routes, WebSocket PTY terminals, LLM provider abstraction, Daytona sandbox management, and MongoDB persistence.

2. Setting Up & Running the Project

This section covers everything you need to clone, configure, and run all parts of the bolt.diy project — backend, desktop frontend, and mobile app — in a local development environment.

2.1 Prerequisites

Before setting up the project, ensure you have the following installed:

🟢 Node.js

Version ≥ 18.18.0 (required by all packages). Recommended: latest LTS (v20+). Check with node --version.

Download: nodejs.org

📦 npm

Bundled with Node.js. Check with npm --version. The project uses npm ci for reproducible installs in CI.

⚠️ The project may also use pnpm, bun, or yarn. Check package.json for packageManager field if npm fails.

☕ Java 17+

Required only for Android builds. The Gradle wrapper uses Java 17 (Temurin recommended).

Use SDKMAN: sdk install java 17.0.9-tem

🛠️ Git

For cloning the repository. Check with git --version.

📱 Android SDK (optional)

Only needed for building the Android APK. Install Android Studio or use sdkmanager CLI. Requires API 34+ and build-tools.

🔑 API Keys (Optional but Recommended)

The following environment variables can be configured. Only the ones you need are required — start with one AI provider.

AI Provider Keys

  • ANTHROPIC_API_KEY — Get key
  • OPENAI_API_KEY — Get key
  • GOOGLE_GENERATIVE_AI_API_KEY
  • GROQ_API_KEY

Infrastructure Keys

  • MONGODB_URI — for chat persistence
  • DAYTONA_API_KEY — for sandboxes
  • GITHUB_TOKEN — for bug reports
⚠️ Running without keys: The backend will start but AI chat requests will fail until at least one AI provider API key is configured.

2.2 Quick Start

The fastest way to get bolt.diy running locally — clones, installs everything, and starts both the backend and desktop frontend.

# 1. Clone the repository
git clone https://github.com/<your-org>/bolt-diy.git
cd bolt-diy

# 2. Install all dependencies (root + backend + frontend)
npm run install:all

# 3. Create .env file for the backend
cat > backend/.env << 'EOF'
# At minimum, set one AI provider key:
ANTHROPIC_API_KEY=sk-ant-...
# Optional persistence:
MONGODB_URI=mongodb://localhost:27017/bolt-diy
# Optional sandbox:
DAYTONA_API_KEY=your-daytona-key
EOF

# 4. Start development servers (backend :3001 + frontend :5173)
npm run dev
✅ Done! Open http://localhost:5173 in your browser. The backend runs on http://localhost:3001.

2.7 Troubleshooting

🟡 Backend won't start

Symptom: Error: Cannot find module

Fix: Run npm --prefix backend install to install backend dependencies.

Symptom: Port 3001 already in use

Fix: Kill the process: lsof -ti:3001 | xargs kill -9, or set PORT=3002 in .env.

🟡 AI Chat not working

Symptom: 401 Authentication Error or No response

Fix: Ensure you have set a valid API key in backend/.env for the provider you're using (e.g., ANTHROPIC_API_KEY, OPENAI_API_KEY).

Symptom: Model list is empty

Fix: Check the backend logs for provider registration errors. Some providers require dynamic model fetching via API.

🟡 MongoDB connection failed

Symptom: MongoDB not configured errors on chat storage endpoints

Fix: Set MONGODB_URI in backend/.env. If you don't need persistence, the server will still work — chat history simply won't be saved.

🟡 Android build fails

Symptom: Gradle build errors or missing SDK

Fix: Verify Java 17+ is installed (java -version). Ensure ANDROID_HOME points to a valid SDK installation with API 34+.

Symptom: npx cap sync fails

Fix: Ensure the mobile frontend built successfully first (npm run build in mobile-frontend/).

🛠️ General Debugging Tips

  • ✦ Check backend logs: The server outputs colored scoped logs (INFO, WARN, ERROR) with scope tags like [Server], [MongoDB], [LLMManager].
  • ✦ Set DEBUG=1 or adjust the log level in backend/src/utils/logger.ts for more verbose output.
  • ✦ Run npm run typecheck from root to check TypeScript compilation across all packages.
  • ✦ Run npm run lint from root to verify code style.
  • ✦ The NODE_ENV environment variable affects logging verbosity — trace/debug level logs are suppressed in production.

4.2 Database & Models

MongoDB is used for chat persistence. Connection is established on server start and gracefully disconnected on shutdown signals (SIGTERM/SIGINT).

MongoDB Connection

File: backend/src/lib/db/mongodb.ts

// Connection flow:
export async function connectMongoDB(): Promise<void> {
  const uri = process.env.MONGODB_URI;
  if (!uri) {
    logger.warn('MONGODB_URI not set — disabled');
    return;
  }
  await mongoose.connect(uri);
  isConnected = true;
}

export function isMongoDBConnected(): boolean {
  return isConnected;
}
⚠️ Conditional: If MONGODB_URI is not set, MongoDB is skipped entirely. The server still functions — chat persistence is simply unavailable.

Chat Schema (Mongoose)

File: backend/src/lib/models/Chat.ts

const ChatSchema = new Schema({

Index: updatedAt: -1 for efficient chat history listing sorted by most recent.

Auto-Cleanup Mechanism

// 30-minute TTL
private readonly SESSION_TTL_MS = 30 * 60 * 1000;

// Checks every 60 seconds
private startCleanupTimer(id: string) {
  setInterval(() => {
    const elapsed = Date.now() - session.lastActivity.getTime();
    if (elapsed > SESSION_TTL_MS) {
      this.destroySandbox(id);
    }
  }, 60_000);
}

// Graceful shutdown
process.on('SIGTERM', async () => {
  await SandboxManager.getInstance().cleanup();
  await disconnectMongoDB();
  process.exit(0);
});

7. Full API Reference

Detailed request/response schemas for the most important API endpoints.

POST /api/chat — Streaming Chat

Streams AI responses via Server-Sent Events (SSE).

// Request Body
{
  "messages": [{ "role": "user"|"assistant"|"system", "content": "string" }],
  "provider": "Anthropic",            // optional, default: Anthropic
  "model": "claude-3-5-sonnet-latest", // optional
  "apiKeys": {},                       // optional
  "providerSettings": {},              // optional
  "systemPrompt": "..."                // optional, overrides default
}

// Response (SSE Stream)
data: {"text": "Hello"}
data: {"text": "! How"}
data: {"text": " can I help"}
data: {"text": " you?"}
data: [DONE]

POST /api/llmcall — Non-Streaming LLM

Returns complete AI response as JSON. Supports chatId for MongoDB system prompt resolution.

// Request Body
{
  "message": "Build me a todo app",     // single message string
  "messages": [...],                     // OR message array
  "chatId": "chat-1719584400000",        // optional, for system prompt lookup
  "provider": "Anthropic",
  "model": "claude-3-5-sonnet-latest",
  "apiKeys": {}
}

// Response
{
  "content": "Here's a complete todo app..."
}

GET /api/chat-storage — List Chats

Returns chat summaries (no full messages) for sidebar display.

// Response

  {
